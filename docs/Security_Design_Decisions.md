# セキュリティ設計判断と運用上の注意

この文書は、sendCMDで意図的に採用しているセキュリティ上の判断、受け入れるリスク、運用で必須となる代替対策を記録します。将来の担当者が、HTTPSやPC別APIキーが単なる実装漏れではないことを判断できるようにするためのものです。

## 1. HTTPSを採用しない

### 判断

本システムではHTTP（既定TCP 5000番）を使用し、HTTPSは採用しません。ローカル認証局、PCごとの証明書発行、有効期限管理、更新、失効、PC交換時の再発行が現在の運用要件に収まらないためです。

### 現在備える保護

- 共有APIキーを直接送信せず、HMAC-SHA256署名を使用する
- 署名対象に時刻、nonce、HTTPメソッド、パス、本文ハッシュ、アップロードファイル名ハッシュを含める
- 使用済みnonceを再起動後も短期間保持し、再送攻撃を拒否する
- 教員PCと生徒PCの時刻差が5分を超える要求を拒否する

### 受け入れるリスク

HMACは要求の認証と改ざん検出を行いますが、通信を暗号化しません。ネットワーク上で通信を観測できる者には、コマンド内容、アップロード内容、取得結果、画面画像などが見える可能性があります。また、HTTP応答自体はTLSで保護されません。

したがって、次の環境では使用しません。

- インターネットへ直接公開された環境
- 来訪者や生徒が自由に接続できる無線LAN
- 信頼できない端末が通信を観測できるネットワーク
- 教員PCから生徒PCへの管理通信を分離できない広域ネットワーク

### 必須の代替対策

- 信頼済みの閉じた教室LANだけで使用する
- TCP 5000番をインターネットへ転送しない
- 可能なら校内ファイアウォールまたはVLANで教員PCから生徒PCへの通信だけを許可する
- 教員PCとネットワーク機器の管理者権限を厳格に管理する
- 機密情報、個人情報、パスワードをPowerShellコマンドや配布ファイルへ含めない

ネットワーク要件が変わり、信頼できない区間を通る場合は、運用開始前にHTTPSまたはVPNなどの暗号化経路を必須とします。

## 2. 全PC共通のAPIキーを維持する

### 判断

教員クライアントと全生徒PCで、意図的に1つの共有APIキーを使用します。PC別キーの発行、対応表、交換、再インストール時の管理負荷を避け、教室単位の運用を単純に保つためです。

### 現在備える保護

- APIキーは16文字以上を強制し、ランダムな32文字以上を推奨する
- 教員PCではWindowsユーザー単位のDPAPIで暗号化する
- 生徒PCではPC単位のDPAPIで暗号化する
- 設定ファイルをAdministratorsとLocal Systemだけに制限する
- APIキーをログへ記録しない
- USBから登録後、生徒PCへ平文キーファイルを残さない

DPAPIとACLは一般ユーザーからキーを保護しますが、そのPCのAdministratorsまたはLocal Systemまで侵害された場合の抽出を完全には防げません。

### 受け入れるリスク

1台の管理者権限、教員PC、またはプロビジョニングUSBから共有キーが漏えいすると、全PCが影響を受けます。漏えい時に1台だけを無効化することはできず、教員クライアントと全生徒PCのキー交換が必要です。

### 必須の代替対策

- APIキーをメール、チャット、通常の共有フォルダーへ置かない
- プロビジョニングUSBをBitLocker To Goで暗号化する
- USBは管理者が携行し、作業後はキーファイルの削除を確認する
- 教員PCを管理者以外へ貸与しない
- 漏えいまたはUSB紛失の疑いがある場合は、漏えいしたものとして全台のキーを交換する
- キー変更中は新旧キーが混在するため、全台完了まで通常の一括操作を止める

## 3. 判断を見直す条件

次のいずれかが発生した場合、この設計判断を見直します。

- 管理通信が信頼できないネットワークを通る
- インターネット越しの管理が必要になる
- 生徒または第三者が教室LANを自由に観測できる
- PC単位のアクセス無効化や監査が必要になる
- 証明書または端末管理を自動化できる基盤が導入される
- APIキー漏えいが繰り返し発生する

見直し時の候補は、HTTPS、VPN、PC別APIキー、端末証明書、集中型の秘密情報管理です。
